Quantify Mobile · Phase 1

Mobile Scope

What goes on the mobile app, what stays on desktop, and why — based on the full Quantify desktop feature set.

Starting point

Desktop Quantify has 14 modules, 6-tab shipment dialogs, 30+ reports, billing, invoicing, pivot tables, and custom properties per company. A new employee takes months to learn it. The mobile app is not a smaller version of this. It is one job: confirm what goes on and off the truck.

The client said it directly: "To me, when I think of a mobile app, I see somebody with a tablet in the yard confirming what went." Every design decision should be tested against that sentence.

Desktop audit — what exists, what it means for mobile

Desktop feature What it does Mobile?
Straight shipment (Way 1) Office creates a DEL instantly with no verification step. Rent starts immediately. Out Office-only action. No counting involved.
Reservation → Release (Way 2) Office creates reservation → yard worker counts what goes on truck → office releases. In The counting step is the core mobile job.
Reservation → Release → Customer confirmation (Way 3) After office releases, customer at job site counts what arrived and confirms independently. Later Valid mobile use case but adds complexity. Phase 2.
Pre-Return counting Office schedules expected return → truck arrives → yard worker counts items off truck. In Same counting job as delivery, opposite direction.
Straight return (no pre-return) Right-click → Return → type quantities → done. No scheduling. Later Requires creating a return on mobile, which means more fields. Start with pre-returns only.
Equipment condition flagging On return: mark items Normal / To Be Serviced / Damaged / Scrapped / Lost. In Yard worker catches this in the moment. Valuable in v1.
Over-returns (negative balances) Worker returns more than expected. System allows it — creates negative balance. In Must handle this. Count screen must allow adding items not on the original shipment.
Barcode / kit scanning Scan individual items or a kit barcode that expands to full product list automatically. Later Tablet cameras are inconsistent. Design for manual counting first. Add scanning in phase 2.
Shipment Properties tab Custom fields per company configured in Global Options (e.g. PO number, driver name). Out Read-only reference at most. Not a yard worker concern.
Products tab (line items) The list of scaffold equipment on the shipment with quantities. In The entire count screen. This is the job.
Consumables tab One-way items that don't come back (netting, sheeting, plastic ties). Out Consumables are set by office. Yard worker doesn't count these — they're buried on site.
Additional Charges tab One-off fees: erection charge, transport, dismantle fee. Out Office billing detail. Zero relevance to yard worker.
Recurring Charges tab Fixed fee billed every 28 days. Sent out "like a product." Out Billing only. Not something a yard worker touches.
Attachments tab Files attached to the shipment. Out Maybe photo-of-damage in phase 2.
Pick ticket printing Printed document with expected quantities and 1–3 signature lines. Out The mobile app replaces this. No printing needed.
Discrepancy resolution Office decides: customer's fault or company's fault. Adjusts quantities accordingly. Out Office decision on desktop. Mobile only flags the discrepancy.
Transfer (TRF) Equipment moves between two yard/branch locations. Later Same counting job. Include in phase 2 once delivery/return is stable.
Reservation (RSV) creation Future-dated hold created by office staff. Out Office creates these on desktop. Mobile only consumes them.
Invoicing / billing Invoice generation, rate profiles, billing methods, accounting exports. Out Entirely office domain. Never show billing data to yard workers.
Reports / pivots 30+ report types. Product history, forecasting, inventory counts. Out Desktop only. Not field work.
Inventory states Available / On Rent / Reserved / In Transit / To Be Serviced / Out of Service. Partial Worker sees "In Transit" and "On Rent" as confirmation feedback. They don't manage states — the system changes them automatically on confirm.
Equipment requests portal Field / customer user requests materials via Quantify Web. Later Different user persona (site supervisor, not yard worker). Separate flow.
Scaffold Register / activities Full scaffold lifecycle — build, inspect, modify, dismantle. Out Separate product area entirely. Not in shipments scope.
QR code scanning (scaffold tags) Anyone scans a QR tag to view scaffold details. Out Scaffold register feature, not shipments.

Screens — v1

Everything the mobile app needs. Nothing more.

1
Sign In
Username + password. Microsoft SSO option. Server URL in footer. Connection settings modal.
From desktop: standard Quantify / Entra ID auth — same credentials
2
Shipments List
Today's deliveries and returns for this branch. Shipment number, job site name, type (DEL/RET), item count, scheduled time. Tap to open.
From desktop: Shipments tab filtered to today + this branch
3
Shipment Detail
Job site name, type, date, driver/vehicle if set. List of products with expected quantities. One tap to start counting.
From desktop: Shipment header fields + Products tab line items only
4
Count Screen
One row per product. Expected quantity shown. Worker taps +/− to enter actual count. Changed rows highlighted. Items not on list can be added. Running discrepancy total visible.
From desktop: Products tab quantities — the part that was previously done on paper (pick ticket)
5
Condition Flag (returns only)
Appears inline per item on returns. Options: Normal / To Be Serviced / Damaged / Scrapped / Lost. Default is Normal — worker only taps if something is wrong.
From desktop: Product condition dropdown on return dialog
6
Confirm Screen
Summary before submitting. Shows any discrepancies clearly. If all match: green. If not: amber with item-level breakdown. Worker confirms or goes back to adjust.
From desktop: the review step before releasing a shipment
7
Success
Clear confirmation. Shipment number. What just happened in one line (e.g. "24 items confirmed — DEL #4821 released"). Back to list.
New — desktop has no equivalent mobile success state

Design decisions this scope forces

No creating shipments on mobile. The yard worker only sees shipments the office already created. If a truck arrives without a pre-return set up, the worker cannot create it on mobile. They call the office. This is a constraint to confirm with Brian.

Products only — no consumables, charges, or recurring items. The count screen shows scaffold equipment (tubes, boards, couplers) only. Consumables and billing items are invisible to yard workers. They don't affect what goes on the truck in a countable way.

Condition flagging applies to the whole row, not a split quantity. This matches the current desktop limitation: you can't mark 2 out of 18 couplers as damaged and 16 as normal on the same return. The client described this as "very annoying." Do not design around this limitation in v1 — confirm with Brian if it should be solved.
