The users

This app is for people working
outdoors with gloves on

Yard workers, delivery drivers, and site supervisors. They use Quantify on job sites — in direct sunlight, with dirty or wet screens, often wearing work gloves. They need to get in, complete a task, and get out. Speed and clarity are everything.

This context should drive every design and development decision we make. This is not a consumer app — it's an operational tool for power users in harsh conditions. I'll walk through the specific UX recommendations I have based on this.

Design system

I took Uno's Material Design 3 library
and rebuilt it for Avontus

Uno ships with standard Material Design components. I customized the entire design system to match Avontus brand guidelines — so what you see in Figma isn't generic Material, it's our visual language.

Switzer typeface

Replaced default Roboto. Two weights: Regular (400) for body, Medium (500) for headings. Tight negative letter-spacing throughout for a sharp, modern feel.

Monochromatic blue

One hue — #0A3EFF. Everything else is pure neutral gray. No gradients, no shadows. Flat and sharp.

0px corners

Zero border-radius on every element — buttons, cards, inputs, modals. This is the defining visual trait of the Avontus brand. Sharp geometry, no softness.

UX recommendations

Three rules I recommend
we follow going forward

56px

Minimum touch target

Every tappable row should be at least 56px tall. Gloved fingers on a wet screen need large, forgiving targets. No small icons, no tiny links.

14px

Minimum readable text

No readable text smaller than 14px — that includes error messages, labels, and helper text. Body text the user acts on should be 16px. Only truly ancillary content like copyright goes to 13px.

#777

No invisible grays

Secondary text should never be lighter than #777777. In direct sunlight, anything lighter disappears. Primary text stays at #111 or #202020, secondary at #555, tertiary at #777.

UX approach

Inline validation, not modals.
Offline-first, not online-only.

Why inline errors matter here

A field worker standing at a scaffold can't deal with a popup listing 7 errors. I designed each error to appear directly under its field — red text, impossible to miss, clear about what's wrong. The form validation screen shows all 7 errors inline so developers can see the pattern.

Why we need offline-first

Job sites have spotty connectivity. I designed the Home screen with both online and offline states. When offline, users see exactly how many changes are waiting — "3 changes queued, will sync when online." The connection settings screen handles error and success states explicitly.

I also recommend keeping cognitive load minimal — the Home screen has exactly 3 navigation options. Login has 2 fields and 2 buttons. These users are busy, they need to get in, complete a task, and get out.

How components work

Each component has fixed properties.
I pick from what exists.

Think of each component as a configurable control with a set of switches. The variant I select in Figma is the exact same variant you render in XAML. No guessing — it's a direct mapping.

Every component works this way — TextBox has Type, State, Label, Populated, Placeholder, Multiline. ToggleSwitch has State and Selected. The properties are fixed, the values are finite.

The challenge

Complex UIs are built by composing
simple parts together

Some elements in the designs aren't a single component — they're layout frames wrapping multiple primitives. Here's how the three most complex elements break down:

Offline indicator Frame + border → Red dot (6px) → Text "Offline" = Status chip
Server URL input Frame "https://" → TextBox (Uno) = Prefixed input
Error banner Frame (red bg) → Text (red, medium) = Error message

In your XAML, you'll do the same — StackPanels and Grids wrapping Uno controls. No custom renderers needed, just smart layout composition.

Looking ahead

What if we need more than
40 components?

Right now, the standard Uno component library covers everything we need. But if we ever need complex custom designs — data visualizations, specialized controls, things Material Design doesn't provide — there are AI-assisted approaches that can read Figma designs and help generate the corresponding XAML.

We're not there yet. But the path exists when we need it.
